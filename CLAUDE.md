# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

`AGENTS.md` (Chinese) and `docs/product.md` are the authoritative product/agent rules; this file adds the architecture and commands. Read `docs/product.md` before touching the teacher or diagnostic.

## Product north star (non-negotiable, enforced in code)

Help students of **all ages** (小学到高中及成人自学者, age 6–99) stop freezing on math **terms**: first understand *what relationship is being talked about* in plain language, then formal textbook wording. Not a problem-solver, not a search tool, not an encyclopedia.

The teaching order is a hard contract, not a prompt slogan:
1. plain-language terms → 2. what it's really about → 3. why it exists → 4. example → 5. textbook definition → 6. self-check.

Concretely this means: never assume the student knows a term; never explain a new term with another unexplained term; on first use, default to the placement diagnostic and otherwise treat the student as zero-background. Tests assert the *section order of the output*, not that a slogan appears in the prompt — keep it that way.

## Commands

Do not run builds/compiles or long verification locally — prefer GitHub Actions (`.github/workflows/ci.yml`). CI runs `ruff check .` then `pytest -q` for Python, and `gradle :android-app:assembleDebug` for Android.

```bash
python -m pip install -e ".[dev]"          # editable install + dev deps
python -m pytest                            # all tests
python -m pytest tests/test_teacher_policy.py            # one file
python -m pytest -k term_first                           # one test by name
python -m ruff check src tests              # lint (E,F,I,UP,B; line length 100)
PYTHONPATH=src uvicorn math_learning_graph.api:app --reload   # local API
python -m math_learning_graph.android_export android-app/src/main/assets/topics.json  # regenerate offline topics
```

## Architecture

Backend-first Python core in `src/math_learning_graph/`, plus a deliberately thin Android shell in `android-app/`. All math/domain logic lives in the backend; the Android app never carries domain logic. No web frontend.

**Data flow:** `seed.load_knowledge_points()` merges three content sources — `curriculum_seed` (primary/junior, 人教版), `high_school_seed`, and `curriculum_expand` — running each `KnowledgePoint` through `depth.with_deep_scaffold` to fill missing layers/examples/practice/reflection. `load_knowledge_points()` **fails loudly if any topic lacks `term_explanations`** (term-first invariant). Those points build a `graph.KnowledgeGraph` (a NetworkX DiGraph; prerequisite edges → topological `learning_order`, ancestors = prerequisites, descendants = future topics). `service.MathLearningService` is the single orchestrator the API depends on; `api.py` is thin FastAPI wiring over it.

**Teacher (`teacher.py`) is the heart.** `build_teacher_prompt` / `build_teacher_answer` assemble the six fixed `SECTION_*` blocks in the mandated order. `answer_section_order()` extracts the realized order and is what tests assert against.

**AI is fallback-guarded, not trusted.** In `service.teacher_answer`, if an OpenAI answer comes back but its section order doesn't equal `TEACHER_SECTION_ORDER`, it is discarded and the deterministic local `build_teacher_answer` is used. The local teacher also runs whenever no `OPENAI_API_KEY`/`OPENAI_MODEL` is configured (`openai_teacher.openai_teacher_from_env`). Uses the OpenAI **Responses API** (`responses.create`), not chat completions.

**Placement diagnostic (`diagnostic.py`):** a term/relationship probe (not an exam) covering rank 1 (等号) through rank 6 (导数). Scoring climbs the ladder rung by rung — a rank only counts once every rank below it is fully confirmed — and only frontier misses (rank ≤ level+1) are recorded as weak, so a beginner failing the derivative probe isn't marked "weak" on it. Unanswered items are unprobed, not weak (old clients submitting fewer items stay compatible). `public_diagnostic_session()` strips correct answers for the client; results flow into teacher prompts as `placement_level`/`placement_summary`. **Lockstep invariant:** `MainActivity.kt` hardcodes the same items (`offlineDiagnosticItems`) and submits raw `choice_index` to the backend scorer — any change to item ids, choices, or correct indices must be mirrored in both files in the same commit.

**Models (`models.py`):** all Pydantic v2, with `GradeBand`/`MathDomain` StrEnums. `review.py` is standalone SM-2-style spaced-repetition scheduling.

**Android (`android-app/`, Kotlin + Compose + Material 3):** single `MainActivity.kt`. Talks to the backend at `http://10.0.2.2:8000` by default (host+port configurable in Settings), and falls back to the bundled `assets/topics.json` (regenerated by `android_export.py`) offline. Forces the placement screen on first run.

## Conventions

- Python 3.11+, keep type hints. `snake_case` for functions/vars/modules, `PascalCase` for Pydantic models/classes. Keep modules small and single-purpose; extend existing ones rather than adding new files/abstractions.
- Test public behavior (graph traversal, profiles, review timing, API responses, teacher section order), not internals. New tests: `tests/test_<behavior>.py`.
- Commits: Conventional-Commits-style imperative prefixes (`feat:`, `fix:`, `test:`).
- Before working on graph/memory/AI-teacher/diagnosis, consult `docs/open-source-reference.md` (MathClaw, math-knowledge-graph, MathVoice as design references — borrow structure, don't clone their products).
